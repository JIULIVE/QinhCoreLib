# QinhCoreLib 规划总纲

> **版本：** 2026.06
> **定位：** 秦淮生态底座与跨插件协议层
> **目标：** 从“能用”进化到“可维护、可接入、可诊断、可替换、可观测”

---

## 1. 使命

QinhCoreLib 不是业务玩法插件，而是秦淮全系的 **平台、桥、经济、脚本、GUI、物品引用标准、诊断中枢**。

它要统一：
- `ItemSource`
- `ItemManagerAPI`
- `CustomGui`
- `GraalJS / ScriptBridge`
- `EconomyBridge`
- `DatabaseManager`
- `PDC`
- `PAPI`
- `Skill bridge`
- 第三方兼容桥

### 核心原则

1. **平台先行**：QCL 负责协议和通用能力，不做业务玩法。
2. **模块可关**：桥和模块都要可开可关。
3. **兼容优先**：MI / NI / QI 迁移期可共存。
4. **对外只暴露 API**：第三方优先走接口，不碰内部实现。
5. **可替换**：GUI、经济、脚本、数据库、ItemSource 都要可替换。
6. **不锁死后续实现**：允许未来替换 Bridge / Provider / Backend。

## 4. 对标 MI / NI 的平台差异

### MI 侧

MI 的强项在于单体成熟、内容丰富。QCL 不抄单体，而是把平台能力外置成可组合组件。

### NI 侧

NI 的强项在于动作系统、脚本表达、生态整合速度。QCL 的策略是：

- 用更干净的 API 替代硬耦合内部类
- 用统一 ItemSource 替代多个专属读取路径
- 用模块化开关替代装上即全开
- 用可测试的桥接层替代隐式反射链

### QCL 的胜负手

QCL 不是与 MI/NI 正面抢“内容量”，而是抢：
- 平台抽象能力
- 迁移友好性
- 可观测性
- 可替换性
- 生态统一协议

## 6. 与各项目的契约边界

| 消费方 | QCL 提供 | QCL 不做 |
|--------|----------|----------|
| QinhItems | ItemSource / Action 桥 / GUI / PAPI | 不写 QI 内核逻辑 |
| QinhSkills | 技能事件 / 脚本 / 物品触发入口 | 不解析技能图本体 |
| QinhStrengthen | GUI / 经济 / 物品源 / 状态桥 | 不负责强化成功率 |
| QinhForge | GUI / 经济 / 物品源 / 桥 | 不负责配方、产出规则 |
| 第三方插件 | API / 注册点 / 兼容桥 | 不强行侵入第三方实现 |
| MythicMobs | 仅作为 QS 的执行后端被诊断和桥接 | QCL 不直接实现技能逻辑 |

## 8. 验收标准

1. 服主只装 QCL 与任意一款秦淮插件，启动零报错
2. `/qcl status` 能短时间显示所有核心桥状态
3. 第三方只依赖 API 即可注册 ItemSource / GUI / Bridge
4. MI / NI 混服场景中，QCL 能稳定处理多源并存
5. 文档可让新开发者在不看源码的前提下完成接入
6. MM 可用性可被统一诊断并反馈给 QS
7. 所有桥接失败都能给出可读原因与建议

---

## 9. 结论

QinhCoreLib 的目标不是“比 NI 更大”，而是做成一个更清晰、更可维护、更适合长期演进的基础平台。当前优先级是：**诊断、API、文档、稳定性、状态统一**。
