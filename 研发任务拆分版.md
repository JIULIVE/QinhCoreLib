# 秦淮生态研发任务拆分版

> 适用范围：`QinhCoreLib`、`QinhItems`、`QinhSkills`、`QinhStrengthen`、`QinhForge`
>
> 目标：把“规划目标”拆成可执行、可排期、可验收的研发任务，方便按阶段推进、按模块分工、按里程碑交付。
>
> 说明：本文件不是愿景文档，而是研发执行清单。每一项都尽量写成“能做什么、先做什么、完成标准是什么”。

---

## 0. 拆分原则

### 0.1 任务拆分原则

1. **先基础、后增强**：先做闭环，再做花活。
2. **先统一、后分化**：先统一协议，再做模块差异。
3. **先可用、后好用**：先保证能跑，再提升体验。
4. **先可诊断、后可优化**：先能定位问题，再谈性能和美学。
5. **先平台、后玩法**：QCL 先打底，业务模块再扩展。

### 0.2 里程碑命名规则

- **P1 / P2 / P3**：产品阶段目标，表示功能推进顺序
- **M1 / M2**：中期里程碑，表示工具链与体验层升级
- **S1 / S2**：专项任务组，表示跨模块联动或技术专项
- **H1 / H2**：高优先级阻断项，表示必须先解决的基础问题

## 2. `QinhCoreLib` 研发任务拆分

### 2.1 H1：必须先做的底座任务

#### H1-1 模块生命周期治理
**目标**：让模块加载、启用、关闭、卸载有明确状态和依赖关系。

**任务拆分**：
- [ ] 为模块定义状态枚举：`Registered / Loaded / Enabled / Disabled / Failed / Unloaded`
- [ ] 为模块增加依赖声明
- [ ] 为模块增加失败隔离机制
- [ ] 为模块增加健康检查入口
- [ ] 为模块增加统一异常包装与上报
- [ ] 为模块增加重载时的资源释放顺序

**验收标准**：
- 单个模块失败不会拖垮整个核心
- 服主能知道哪个模块出问题、为什么出问题
- 模块重载后不会遗留明显脏状态

#### H1-2 统一诊断入口
**目标**：做出 `/qcl status` 或等价诊断能力。

**任务拆分**：
- [ ] 输出核心版本信息
- [ ] 输出插件依赖状态
- [ ] 输出桥接状态
- [ ] 输出数据库状态
- [ ] 输出脚本运行状态
- [ ] 输出物品来源状态
- [ ] 输出经济状态
- [ ] 输出 Placeholder 状态

**验收标准**：
- 一条命令就能看到当前系统关键状态
- 所有失败项都带原因和建议

#### H1-3 统一错误模型
**目标**：所有桥、服务、模块返回统一错误对象。

**任务拆分**：
- [ ] 设计标准错误码
- [ ] 设计错误来源字段
- [ ] 设计是否可恢复字段
- [ ] 设计建议字段
- [ ] 所有桥接返回值统一包装

**验收标准**：
- 不再只返回 null 或布尔值
- 每个失败都能定位到来源和处理建议

## 9. 研发验收标准总表

| 层级 | 验收标准 |
|------|----------|
| 功能可用 | 核心闭环能跑 |

## 10. 结语

这个“研发任务拆分版”的核心价值，不是把目标写得更宏大，而是把目标变成可以执行的研发清单。只要按照 P1 → P2 → P3 → M1 → M2 的节奏推进，Qinh 就能从“有想法”走向“有产品”，再走向“有生态”。
